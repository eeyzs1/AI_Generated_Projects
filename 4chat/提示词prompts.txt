包含以下功能模块：
1. 用户注册和登录系统：用户可以注册新账户并登录。
   - 用户模型包括用户名、显示名称（displayname）、密码（加密存储）、邮箱、头像等字段。
   - 注册接口验证用户名唯一性，登录接口验证用户名和密码匹配。
   - 未注册用户需要先注册才能登录，未登录用户需要先登录才能使用后续功能
   - 注册时，用户需要设置displayname（在聊天时显示的名称，允许重复），并从默认头像列表中选择一个头像，或上传自己的头像。
   - 头像上传功能：
     - 用户上传头像后，会显示预览图
     - 用户可以使用圆形选择框选择要截取的区域
     - 选择框可以拖动和调整大小,当用户鼠标位于图像区域内时,滑动滚轮应该缩放选择框而不是滚动整个页面，按住鼠标左键可以拖动整个选择框，按住选择框四角的选择点进行拖动可以进行缩放
     - 点击确认按钮后，会裁剪选择框内的区域,并将其作为用户在前端选择的头像
     - 头像会被缩放到200x200大小的圆形头像
   - 头像上传限制：大小不超过2MB，尺寸建议为200x200像素，支持常见图片格式（JPG、PNG、GIF）。
   
2. 聊天室功能：用户可以创建聊天室并邀请其他用户加入。
   - 聊天室模型包括房间名、创建者ID、成员列表等字段。
   - 创建房间接口需要验证用户身份。
   - 邀请功能：用户可以邀请其他用户加入聊天室，被邀请者可以选择接受或拒绝邀请。
   - 被邀请者接受邀请后自动成为聊天室成员，可参与聊天。
   
3. 消息发送和接收：用户可以在聊天室内发送和接收文本消息。
   - 消息模型包括发送者ID、接收者ID/聊天室ID、消息内容等字段。
   - 发送消息接口需要验证用户是否属于该聊天室。
   - 当一个用户发送消息时，同一聊天室的所有用户都能实时接收到消息，消息中应显示发送者的displayname和头像。
   - 新消息应显示在聊天记录的下端，而不是上端。
   
4. 在线用户列表：实时显示当前在线用户的列表。
   - 使用WebSocket实现实时更新在线用户列表。
   - 在线用户列表应显示用户的displayname和头像。
   
5. 用户中心页面：用户可以查看和修改自己的个人信息。
   - 用户可以查看自己的用户名、显示名称（displayname）、邮箱、账户创建时间、头像等信息。
   - 用户可以修改自己的用户名、显示名称（displayname）、邮箱和密码。
   - 修改信息时需要验证新用户名和邮箱的唯一性，displayname允许重复。
   - 用户可以修改自己的头像：可以从默认头像列表中选择，或上传新的头像。
   - 头像上传功能：
     - 用户上传头像后，会显示预览图
     - 用户可以使用圆形选择框选择要截取的区域
     - 选择框可以拖动和调整大小,当用户鼠标位于图像区域内时,滑动滚轮应该缩放选择框而不是滚动整个页面，按住鼠标左键可以拖动整个选择框，按住选择框四角的选择点进行拖动可以进行缩放
     - 点击确认按钮后，会裁剪选择框内的区域,并将其作为用户在前端选择的头像
     - 头像会被缩放到200x200大小的圆形头像
   - 头像上传限制：大小不超过2MB，尺寸建议为200x200像素，支持常见图片格式（JPG、PNG、GIF）。
   - 用户可以从用户中心返回聊天室。

6. 邮箱验证功能：用户注册时需要验证邮箱。
   - 用户注册后，后端发送一封带有激活链接的邮件。
   - 激活链接包含验证令牌，保存在数据库中一段时间。
   - 用户点击链接后，账户被激活，否则账户注册失败。

7. 密码重置功能：用户可以找回忘记的密码。
   - 用户在登录页面点击"忘记密码"链接。
   - 用户输入注册邮箱，后端发送密码重置链接。
   - 用户点击链接进入密码重置页面，设置新密码。

8. 联系人功能：用户可以添加和管理联系人。
   - 联系人模型包括发起者ID、接收者ID、状态（待确认、已确认）等字段。
   - 用户可以搜索其他用户并发送联系人请求。
   - 接收联系人请求的用户可以选择接受或拒绝。
   - 联系人关系建立后，双方互为联系人。
   - 用户可以查看自己的联系人列表。
   - 用户可以删除联系人。

9. 通讯录功能：用户可以从通讯录中找到联系人并查看其信息。
   - 通讯录页面显示用户的所有联系人列表，按字母顺序排序。
   - 每个联系人条目显示用户的displayname和头像。
   - 点击联系人条目进入联系人个人信息页面。
   - 联系人个人信息页面显示联系人的displayname和头像。
   - 联系人个人信息页面提供发送消息按钮，点击后进入只有当前用户和联系人的聊天室。
   - 如果聊天室不存在，系统会自动创建一个新的双人聊天室。

技术细节要求：
- 使用FastAPI作为Web框架。
- 使用SQLAlchemy ORM操作MySQL或者sqlite数据库存储用户信息和聊天记录。
- 前端界面使用React（TSX）编写，确保基本的响应式设计。
- 使用WebSocket实现实时消息传递，前端使用WebSocket API连接到后端服务，优化连接管理以提高稳定性。
- 集成JWT（JSON Web Tokens）用于用户身份认证，实现双令牌（Access Token + Refresh Token）机制。
- Access Token存储在内存中，用于短期访问，过期时间较短。
- Refresh Token存储在HttpOnly Cookie中，用于长期刷新，过期时间较长。
- 实现令牌自动刷新机制，当Access Token过期时自动使用Refresh Token刷新。
- 实现令牌撤销功能，支持主动登出和令牌失效。
- 添加Dockerfile支持容器化部署。
- 实现文件上传功能，用于处理用户头像上传，包括文件大小和格式验证。
- 提供默认头像列表，供用户注册和修改头像时选择。
- 头像存储：可以存储在本地文件系统或云存储服务中，并在数据库中保存头像的访问路径。

4. 微服务架构转型要求：将项目从单体应用转型为微服务架构，参考 IM_microservice_design.md。
   - 按业务域拆分为 6 个独立微服务，每个服务独立部署、独立数据库、独立 Dockerfile：
     - user-service（端口 8001）：用户注册/登录/资料管理/联系人管理/Token 管理/邮箱验证/密码重置
       迁移来源：services/auth_service.py、services/contact_service.py、models/user.py、models/contact.py、models/refresh_token.py
     - group-service（端口 8002）：聊天室创建/管理/成员管理/邀请处理
       迁移来源：services/chat_service.py（rooms/invitations 部分）、models/room.py
     - message-service（端口 8003）：消息收发/历史记录存储/分页查询
       迁移来源：services/chat_service.py（messages 部分）、models/message.py
     - connector-service（端口 8004）：WebSocket 长连接维护/实时消息推送/在线状态管理
       迁移来源：services/ws_service.py、main.py 中的 WebSocket 端点
       注意：在线状态存储需从内存改为 Redis，支持多实例水平扩展
     - push-service（端口 8005）：离线消息推送（新建服务，单体中无对应代码）
     - storage-service（端口 8006）：文件/头像上传/裁剪/下载
       迁移来源：main.py 中的头像上传端点、static/avatars/ 目录
   - 中间件要求：
     - MySQL（端口 3306）：各服务独立数据库（im_user / im_group / im_message / im_storage），数据自治，不跨服务直接操作数据库
     - Redis（端口 6379）：存储登录态、在线状态、未读消息数缓存
     - Kafka（端口 9092）：异步事件总线，核心 Topic 包括 msg_sent / user_online / user_offline / group_created / member_changed / user_registered
     - Nacos（端口 8848）：配置中心 + 服务注册发现，所有服务启动后注册到 Nacos，服务间调用通过 Nacos 查地址
     - APISIX（端口 8080）：API 网关，统一路由、鉴权、限流、跨域处理
   - 服务间通信：
     - 同步（HTTP）：用于需要即时结果的权限校验场景
       示例：message-service 发消息前 HTTP 调用 group-service 校验用户是否为群成员
       示例：connector-service 建立连接时 HTTP 调用 user-service 校验 Token 有效性
     - 异步（Kafka）：用于消息分发和事件通知场景
       示例：message-service 发布 msg_sent → connector-service 消费并推送给在线用户
       示例：message-service 发布 msg_sent → push-service 消费并推送给离线用户
   - 网关路由配置（APISIX）：
     - /api/user/* → user-service:8001
     - /api/group/* → group-service:8002
     - /api/message/* → message-service:8003
     - /ws/* → connector-service:8004
     - /api/storage/* → storage-service:8006
   - 部署要求：
     - 编写根目录 docker-compose.yml，一键启动所有微服务和中间件
     - 每个微服务编写独立的 Dockerfile（基于 Python 3.11，FastAPI + uvicorn 启动）
     - 编写 gateway/apisix.yaml 配置 APISIX 路由规则
     - 服务间通过 Docker 内置 DNS 通信（如 user-service:8001），不硬编码 IP
   - 前端适配：
     - 所有 HTTP API 调用改为通过 APISIX 网关统一入口（http://localhost:8080）
     - WebSocket 连接改为 ws://localhost:8080/ws/connect
     - 前端代码迁移到根目录 frontend/（与各微服务平级）
